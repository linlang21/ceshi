## 详细方案

### 1. 🔴 冻结定时器复用进程句柄

**位置：** `WriteMemCoord` 入口 + `CoordFreezeTick` 内部

**现状：**
- `WriteMemCoord` 每次都调用 `OpenGameProcessForCoord` 开新句柄
- `CoordFreezeTick` 在已有 `lockCoordHProcess` 的情况下持续写入 60 帧（约 1 秒）
- 每次启动冻结 = 1 次 OpenProcess；冻结结束后 1 次 CloseHandle，单次传送开销可控
- 但若连续多次传送或与读取坐标交替操作，每次都重新 OpenProcess 浪费

**优化方案：**
- 新增长驻句柄字段 `IntPtr persistentCoordHProcess`，首次打开后保持
- `WriteMemCoord` / `ReadMemCoord` 入口检测长驻句柄是否仍有效（`GetExitCodeProcess` 或 PID 校验）
- 失效时重新打开
- 工具退出 / 游戏关闭时显式 CloseHandle

**收益：** 减少 60+ 次 OpenProcess/CloseHandle 系统调用
**风险：** 句柄长驻可能错过游戏重启事件，需配合 PID 校验

---

### 2. 🔴 AOB 扫描改为后台执行

**位置：** `AOBScanStaticOffset`

**现状：**
- `ReadProcessMemory` 一次性读取 64MB 在 UI 线程同步执行
- 第一次 AOB 扫描时 UI 卡顿明显（数百毫秒）
- 由于硬编码偏移通常有效，AOB 扫描极少触发，但一旦触发体验差

**优化方案（任选其一）：**
- **方案 A（推荐）：** 用 `Task.Run` 后台扫描，扫描期间 `ResolveCoordBaseByPtrChain` 直接返回 false；扫描完成后通过 `BeginInvoke` 回到 UI 线程更新 `cachedStaticOffset`，下次解析直接用
- **方案 B：** 分块读取（如 4MB/块），块间 `Application.DoEvents()` 让 UI 响应
- **方案 C：** 缩小扫描范围。17 处引用分布在 `0x41A3860 ~ 0x438FB67`（约 2MB），可优先扫 8MB 区段；找不到再扩展

**收益：** UI 不卡顿
**风险：** 方案 A 期间传送按钮会短暂失效（指针链未就绪），需 UI 提示

---

### 3. 🔴 CoordFreezeTick 句柄空校验

**位置：** `CoordFreezeTick`

**现状：**
- `coordFreezeCount >= 60` 后调用 `CloseHandle(lockCoordHProcess)`
- 但 `lockCoordHProcess` 字段未重置为 `IntPtr.Zero`
- 若定时器因信号延迟多触发一次 tick，会进入分支调用 `WriteDouble(已关闭句柄, ...)` 抛异常
- 当前用 `try/catch` 兜底但未日志记录，问题不可见

**优化方案：**
```csharp
void CoordFreezeTick(object sender, EventArgs e) {
    coordFreezeCount++;
    if (lockCoordHProcess == IntPtr.Zero) return;  // 新增首行守卫
    try { ... }
    if (coordFreezeCount >= 60) {
        coordFreezeTimer.Stop();
        try { CloseHandle(lockCoordHProcess); } catch { }
        lockCoordHProcess = IntPtr.Zero;  // 新增重置
    }
}
```

**收益：** 避免潜在崩溃
**风险：** 无

### 4. 🟡 ResolveCoordBaseByPtrChain 结果缓存

**位置：** `ResolveCoordBaseByPtrChain`

**现状：**
- 传送按钮高频点击或冻结定时器每 16ms 一次 tick 都重新走完整三层解指针链
- 每次 = 4 次 `ReadInt64`/`ReadDouble` 系统调用

**优化方案：**
- 新增 `long cachedObjBase; DateTime cachedObjBaseExpireAt;`
- 缓存有效期 500ms（游戏内对象指针短时间内不会变）
- 入口先检查缓存是否有效：`DateTime.Now < cachedObjBaseExpireAt && cachedObjBase != 0`
- 缓存命中时直接返回 `cachedObjBase`，跳过所有 RPM
- 失效或读取异常时重新解析

**收益：** 高频操作下 RPM 调用减少 ~95%
**风险：** 玩家被传送后 500ms 内坐标基址若被游戏回收重分配，可能写到错误地址 — 实际玩家对象在会话期间稳定，风险低

### 5. 🟡 AOB 候选提前剪枝

**位置：** `AOBScanStaticOffset` 后半段

**现状：**
- 对所有引用≥3次的偏移都跑完整指针链验证
- 每个 candidate = 4 次 RPM（P1/P2/OBJ/X）
- 若代码段里有大量 RIP-relative 引用同一偏移，验证开销累积

**优化方案：**
- 排序后只取前 N 个候选（如 N=5）做验证
- 或在验证前加一层"地址形态预筛"（P1 必须是堆地址、不在模块区段内）后再跑完整链

**收益：** AOB 扫描耗时减少
**风险：** 极端情况下 N 取太小可能漏掉正确偏移；建议 N=5 起步，必要时调大

---

### 6. 🟡 评估删除 Hook 后备分支

**位置：** `ResolveCoordBase` / `ReadMemCoord` / `WriteMemCoord` / `CoordFreezeTick`

**现状：**
- 静态指针链方案已稳定可用（实测通过）
- Hook 后备分支保留约 50 行（`coordHookStoreAddr`/`coordHookActiveSlot` 相关）
- 三处 `if (lastResolveWasPtrChain) ... else if (coordHookActiveSlot >= 10) ... else ...` 复杂度高

**评估要点：**
- 指针链方案已通过 4 个传送点实测
- AOB 扫描兜底覆盖游戏更新场景
- Hook 后备的唯一价值：指针链中间节点被反作弊改写时 — 半单机模式几乎不会发生
- 删除后维护成本显著降低

**优化方案：**
- 先在 v38.2 标记 Hook 后备为 deprecated，日志中记录"后备分支启用"事件
- 观察 1-2 周无后备触发记录后，在 v38.3 彻底删除
- 保留 `coordHookStoreAddr` 字段定义以防回滚，但移除使用点

**收益：** 代码简化，维护成本下降
**风险：** 若反作弊策略变更导致指针链失效，需要紧急回滚

---

### 7. 🟢 cachedModuleBase 失效保护

**位置：** `ResolveCoordBase` 入口

**现状：**
- `cachedModuleBase` 在首次解析时缓存 yysls.exe 基址
- 玩家重启游戏后模块基址可能变（虽然 ASLR 间距通常一致）
- 当前不校验，若基址失效，所有后续 RPM 都会读取错误地址

**优化方案：**
- 入口检测 `cachedModuleBase` 是否仍指向 yysls 模块
- 简单实现：调用 `GetModuleHandle` / 重新走 `CreateToolhelp32Snapshot` 校验
- 失效时清空 `cachedModuleBase` / `cachedStaticOffset` / `cachedObjBase`，重新初始化

**收益：** 长时间运行工具 + 游戏重启场景下稳定
**风险：** 校验本身有开销，建议只在指针链解析失败时触发

---

### 8. 🟢 坐标范围检查可配置化

**位置：** `ResolveCoordBaseByPtrChain` 校验段 / `AOBScanStaticOffset` 验证段

**现状：**
- 硬编码 `Math.Abs(x) > 100000` 作为坐标合理性上限
- 当前地图够用，但未来大地图扩展可能误判

**优化方案：**
- 抽出为常量 `const double COORD_SANITY_MAX = 100000.0;`
- 或读 `config.txt` 配置项
- 极端情况下可调

**收益：** 可维护性
**风险：** 无
